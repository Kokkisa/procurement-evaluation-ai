# ADR-0006: Hybrid PDF Extraction with OCR Fallback

## Status

Accepted

## Context

Real-world procurement bid packs are heterogeneous. Vendors typically submit:

- **Digital PDFs** for documents they own end-to-end — bidder response forms, declarations on letterhead, internal cover sheets. These have a working text layer; pdfplumber + pypdf extract them cleanly.
- **Notarised scans** for documents they need a third party to attest — audited turnover certificates from a CA firm, GST registration certificates, similar-work experience letters from past clients, no-deviation declarations on stamped paper. These are scanned image-only PDFs whose text layer comes back empty.

Block 4's synthetic test fixtures are 100% digital because they're generated by ReportLab. The Block 5–6 LLM evaluation pipeline was built and validated against those. But on real bid packs, ~30–50% of the per-vendor PDFs are scans. Without OCR, the LLM evaluator sees an empty document and emits `NOT_PROVIDED`, downgrading the verdict for documents that are actually present.

A real example surfaced during Block 13 prep: a vendor's notarised turnover certificate (signed by SURESH SIVA & Co., Chartered Accountants) returned `0 chars` from pdfplumber. The text was there visually — the LLM just couldn't see it.

## Decision

Add a per-page OCR fallback to the existing `extract_text()` in `src/proceval/ingestion/pdf_parser.py`:

1. Open the PDF with pdfplumber as before.
2. For each page, attempt text-layer extraction (body + tables).
3. If the per-page text comes back below `settings.ocr_fallback_threshold` chars (default **50**), invoke `pdf2image` (which wraps Poppler) to rasterise that page at `settings.ocr_dpi` (default **300**), then `pytesseract.image_to_string()` against the rendered image.
4. Run the OCR'd text through `normalize_ocr_text()` to strip mojibake (`â€"`, `Â©`, decorative bullets), collapse whitespace runs, and drop blank lines.
5. Combine text-layer pages and OCR'd pages into the final blob.

The hybrid path is **opt-out** via `OCR_ENABLED=false` (e.g. for the test suite, where ReportLab-generated fixtures are tiny and would all trip the threshold).

## Why hybrid (not always-OCR)

We rejected always-OCR for three reasons:

| Concern | Always-OCR cost | Hybrid cost |
|---|---|---|
| **Latency** | ~2–4 seconds per page on 300 DPI Tesseract | Only the scan pages (~30–50% in practice) |
| **Accuracy on digital PDFs** | OCR loses table structure that pdfplumber preserves; numbers like `38.42 LAKHS` can OCR as `38,42 LAKHS` or `3842 LAKHS` | Digital pages keep their original text layer + table TSV |
| **CPU load** | Continuous Tesseract usage scales linearly with vendor count | Bounded to actual scan pages |

Hybrid keeps the Block 5–6 critical path (digital fixtures) at ~milliseconds per page and only pays OCR cost when the document actually needs it.

## Why this threshold (50 chars)

50 chars per page comfortably separates "thin text-layer / scan / decorative cover page" from "real body text". A typical scanned page with no extractable text returns 0–5 chars (page number plus stray header). A real text-layer page even with one substantive paragraph returns several hundred. The default value catches the failure mode without false-positiving on sparse legitimate pages, and it's a knob (`OCR_FALLBACK_THRESHOLD`) so deployments with unusual bid pack shapes can tune.

## Why aggressive non-ASCII stripping in `normalize_ocr_text`

Tesseract's English language model occasionally hallucinates Unicode characters from page noise — borders, stamps, logos, signature smudges. Common output: `â€"` (mojibake for em-dash), `Â©` (copyright as Latin-1-misread-as-UTF-8), `•` (bullet). Stripping all non-ASCII is a deliberate trade:

- **Win:** OCR'd vendor text downstream of the LLM is uniformly ASCII; no encoding-related prompt drift; no mojibake leaking into the final PDF audit trail.
- **Loss:** Legitimate non-English content (Hindi names, Devanagari script, ₹ rupee symbol) gets stripped too. For v0.1's English-language demo this isn't a concern; for a production deployment with Hindi tender documents we'd swap to Unicode-normalisation + curated regex.

## Consequences

### Positive

- The system handles real bid packs, not just the synthetic fixtures.
- Per-page method is logged at INFO so the operator (and LangSmith trace reader) can see exactly which pages were OCR'd.
- The threshold + DPI + on/off toggle are all `.env`-tunable; ops can dial throughput vs accuracy without code changes.
- `normalize_ocr_text()` is a pure function — easy to unit-test, easy to swap for a smarter normaliser later.

### Negative

- Tesseract becomes a system-level dependency. Anyone running the full pipeline needs `tesseract-ocr` and Poppler installed. We document this in the README's Prerequisites section but don't auto-install. This is acceptable because:
  - Both are MIT/Apache-licensed and on every package manager (`apt install tesseract-ocr poppler-utils` on Debian/Ubuntu, `winget install UB-Mannheim.TesseractOCR` + Poppler-for-Windows release zip).
  - The OCR path is opt-out (`OCR_ENABLED=false`) — the system still runs without the binaries, just with reduced fidelity on scans.
- OCR adds latency on scanned pages. At 300 DPI, ~2–4 seconds per page. A 5-vendor evaluation with 3 scanned pages per vendor adds ~30–60 seconds to the wall-clock runtime of `/confirm`. Acceptable for an audit-grade workflow; not acceptable for sub-second user-facing responses (we're not).
- OCR is not always-correct. ~95% accuracy is realistic on clean scans, less on poor scans. The LLM evaluator doesn't know the input was OCR'd; if a number gets OCR'd wrong, the verdict can be wrong. Mitigation: the human-in-the-loop review step (`POST /review/reject`) catches systematic errors and re-runs with feedback.

## Alternatives Considered

### AWS Textract

Higher accuracy than Tesseract on tables and forms, especially handwritten content. Native to the AWS ecosystem, comes with structured field extraction (key-value pairs, tables as JSON).

**Rejected for v0.1** because:
1. It's a paid API (~$0.0015 per page for Textract Forms) — adds cost on every run.
2. Requires AWS credentials in the deployment, which the v0.1 demo deliberately avoids (we're Anthropic-tier-bound, not AWS-bound).
3. Network round-trip on every scanned page; Tesseract is local-CPU.

Documented as a v0.3+ option for a deployment where Textract's accuracy on handwritten content (signatures, hand-stamped seals) becomes load-bearing.

### Google Cloud Vision OCR

Similar trade-off to Textract. Documented as a v0.3+ alternative.

### Tesseract via Docker sidecar

Avoids host-level Tesseract install. Adds container orchestration complexity for a pure-CPU side workload. Not worth it for v0.1; revisit if the deployment moves to Kubernetes.

### Always run OCR, never trust the text layer

Rejected — see "Why hybrid (not always-OCR)" above. Loses pdfplumber's table-structure preservation for digital docs.

## Implementation

- Code: `src/proceval/ingestion/pdf_parser.py` — `extract_text()` + `_extract_page()` + `_ocr_page()` + `normalize_ocr_text()`.
- Config: `src/proceval/config.py` — `ocr_enabled` / `ocr_fallback_threshold` / `ocr_dpi`.
- `.env.example` documents the three knobs.
- Tests: `tests/test_ocr_extraction.py` — pure-function `normalize_ocr_text` parametrised tests, threshold-logic tests with `pytesseract.image_to_string` mocked, per-page method-logging tests, and a gated live-binary integration test that skips when Tesseract isn't reachable.
- `tests/conftest.py` sets `OCR_ENABLED=false` for the test session so the existing tiny ReportLab fixtures don't trip the threshold; OCR-specific tests re-enable via `monkeypatch.setattr(settings, "ocr_enabled", True)`.
- `scripts/check_ocr.py` — ad-hoc CLI for verifying the hybrid path on an external PDF without committing the file to the repo.

## Verification

Tested against a real notarised turnover certificate (one-page scanned PDF, ~1.2 MB):

```
page 1: text layer thin (0 chars < threshold 50) -> OCR
pages         : 1
total chars   : 1024
```

OCR returned readable structured content (CA firm letterhead, partner names, phone numbers, office addresses). The exact figures we'd want the LLM to extract (turnover, period, signatory) appear correctly in the OCR output.
